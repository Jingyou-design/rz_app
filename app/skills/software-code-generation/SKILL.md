# 软件代码生成

在工作区根目录完成代码文件。工作区中已有说明书 DOCX、Word 模板和填报 TXT，它们均为输入材料，不得修改、删除或移动。

## 工作步骤

1. 先基于说明书确定 6 至 8 个业务模块的固定清单。第一个必须为 `shared_core.py`；清单确定后不得新增、删除或改名模块。
2. 明确每个模块公开的类、函数、输入参数、返回值以及依赖关系。模块只能依赖先前已经生成的模块，禁止循环依赖。
3. 按清单逐个生成模块。每个模块只调用一次 `write_file` 写入完整源码；生成后读取已依赖的真实代码，使用其真实类名、函数名和参数名；不要臆造接口。
4. 每个模块围绕说明书中的一个核心功能实现完整逻辑，目标约 500 行，以实现总量约 3000 行的业务代码。
5. 写完清单中的最后一个模块后，立即调用最终回复结束任务。不要自行统计行数、遍历目录或反复检查已有代码。

## 代码要求

- 只生成核心业务与算法逻辑，不生成 Web 路由、命令行入口、数据模型、测试代码或演示代码。
- 优先使用稳定、通用的第三方库；不得为了凑行数堆砌空分支、重复包装或无意义代码。
- 代码必须能体现说明书中的真实功能流程，包含必要的异常处理、配置入口和模块协作。
- 注释少且简洁，只能使用 `# 简短说明` 形式；禁止使用 `─`、`═`、`-` 等字符拼接的装饰性分隔线、框线或标题。
- 禁止 `TODO`、`FIXME`、`pass`、`eval`、`exec`、伪代码和占位实现。
- 禁止创建统计、校验、临时、辅助或脚本性质的 Python 文件；只允许创建预先确定的业务模块。
- 不要创建子目录，不要修改非 Python 输入材料，不要向用户索要补充信息；根据说明书作出合理实现选择。

## 完成标准

工作区根目录仅新增清单中的 6 至 8 个 `.py` 文件；它们之间的依赖有序、接口一致。最终回复只说明已生成的模块名称。
